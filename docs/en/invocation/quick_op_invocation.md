# Operator Invocation

## Prerequisites

- Environment deployment: Before invoking an operator, complete the basic environment setup by following [Environment Deployment](../context/quick_install.md).
- Operator invocation list: The operators available for invocation in the project are listed in [Operator List](../../zh/op_list.md), and the corresponding aclnn interfaces are listed in [aclnn List](../../zh/op_api_list.md).
- build.sh: Operator invocation depends on the build.sh script in the root directory. You can view the functions through the `bash build.sh --help` command. For parameter descriptions, refer to [build Parameter Description](../context/build.md).

If your compilation environment can access the network, refer to [Online Compilation](#online-compilation). The compilation script automatically downloads the third-party software. Otherwise, refer to [Offline Compilation](#offline-compilation) to manually download the third-party software.

After preparing the open-source third-party software, you can choose from the following compilation methods:

- **Custom operator package**:

  A package generated by compiling selected operators is called a custom operator package. It is applied to the CANN package in a **mounting** manner and does not change the original package content. The generated custom operator package has higher priority than the original CANN package. This package supports aclnn and graph mode invocation of AI Core operators.

- **ops-nn package**:

  A package generated by compiling the entire project is called an ops-nn package. It can **completely replace** the corresponding part of the CANN package. This package supports aclnn and graph mode invocation of AI Core operators.

- **ops-nn static library**:

  > Note: If you need to **perform secondary publishing based on this project** and have **requirements for the software package size**, it is recommended to use static library compilation. This library can be linked to your application development program and only retains the operators required by the business, thereby achieving minimal software deployment.

  This means the entire project is compiled into a single static library file, including libcann_nn_static.a and aclnn interface header files. This package only supports aclnn invocation of AI Core operators.

### Online Compilation

#### Custom Operator Package

1. **Compile the custom operator package**

    Enter the project root directory and execute the following compilation command:

    ```bash
    bash build.sh --pkg --soc=${soc_version} [--vendor_name=${vendor_name}] [--ops=${op_list}]
    # Example: Compile the TransposeBatchMatMul operator
    # bash build.sh --pkg --soc=ascend910b --vendor_name=transpose_batch_mat_mul --ops=transpose_batch_mat_mul
    # Compile user operators in the experimental contribution directory
    # bash build.sh --pkg --experimental --soc=ascend910b --ops=${experimental_op}
    ```

    - --soc: ${soc_version} represents the NPU model. For Atlas A2 series products, use "ascend910b" (default). For Atlas A3 series products, use "ascend910_93". For Ascend 950PR/Ascend 950DT products, use "ascend950".
    - --vendor_name (optional): ${vendor_name} represents the name of the custom operator package to build. The default name is custom.
    - --ops (optional): ${op_list} represents the operators to compile. If not specified, all operators are compiled by default. The format is "transpose_batch_mat_mul,gemm,...", with multiple operators separated by commas.
    - --experimental (optional): Indicates compiling operators in the experimental contribution directory. ${experimental_op} is the directory name of the new contributed operator. For contribution instructions, refer to the [Contribution Guide](../../../CONTRIBUTING.md).

    If neither ${vendor_name} nor ${op_list} is provided, the ops-nn package is compiled. To compile a custom operator package for all operators, you must provide ${vendor_name}. When the following message appears, the compilation is successful.

    ```bash
    Self-extractable archive "cann-ops-nn-${vendor_name}-linux.${arch}.run" successfully created.
    ```

    After successful compilation, the run package is stored in the build_out directory of the project root directory.

2. **Install the custom operator package**

    ```bash
    ./cann-ops-nn-${vendor_name}-linux.${arch}.run
    ```

    The custom operator package installation path is `${ASCEND_HOME_PATH}/opp/vendors`. ${ASCEND_HOME_PATH} is configured through the environment variable and represents the CANN toolkit package installation path, typically ${install_path}/cann.

3. **(Optional) Uninstall the custom operator package**

    After the custom operator package is installed, an `uninstall.sh` script is generated in the `${ASCEND_HOME_PATH}/opp/vendors/${vendor_name}_nn/scripts` directory. Execute this script to uninstall the custom operator package. The specific command is as follows:

    ```bash
    bash ${ASCEND_HOME_PATH}/opp/vendors/${vendor_name}_nn/scripts/uninstall.sh
    ```

#### ops-nn Package

1. **Compile the ops-nn package**

   Enter the project root directory and execute the following compilation command:

   ```bash
   # Compile all operators except those in the experimental contribution directory
   bash build.sh --pkg --soc=${soc_version}
   # Compile all operators in the experimental contribution directory
   # bash build.sh --pkg --experimental --soc=${soc_version}
   ```

   - --soc: ${soc_version} represents the NPU model. For Atlas A2 series products, use "ascend910b" (default). For Atlas A3 series products, use "ascend910_93". For Ascend 950PR/Ascend 950DT products, use "ascend950".
   - --experimental (optional): Indicates compiling operators in the experimental contribution directory.

   If the following message appears, the compilation is successful.

   ```bash
   Self-extractable archive "cann-${soc_name}-ops-nn_${cann_version}_linux-${arch}.run" successfully created.
   ```

   ${soc_name} represents the NPU model name, which is ${soc_version} with "ascend" removed. After successful compilation, the run package is stored in the build_out directory.

2. **Install the ops-nn package**

    ```bash
    # Installation command
    ./cann-${soc_name}-ops-nn_${cann_version}_linux-${arch}.run --full --install-path=${install_path}
    ```

    ${install_path}: Represents the specified installation path, which must be the same path as the toolkit package installation. The default installation directory is `/usr/local/Ascend`.

3. **(Optional) Uninstall the ops-nn package**

    ```bash
    # Uninstallation command
    ./${install_path}/cann/share/info/ops_nn/script/uninstall.sh
    ```

#### ops-nn Static Library

> Note: The static library only supports Atlas A2 and Atlas A3 series products. Experimental operators do not support the static library at this time.

1. **Compile the ops-nn static library**

   Enter the project root directory and execute the following compilation command:

    ```bash
   bash build.sh --pkg --static --soc=${soc_version}
    ```

   ${soc_version} represents the NPU model. For Atlas A2 series products, use "ascend910b" (default). For Atlas A3 series products, use "ascend910_93".

   If the following message appears, the compilation and compression are successful.

    ```bash
   [SUCCESS] Build static lib success!
   Successfully created compressed package: ${repo_path}/build_out/cann-${soc_name}-ops-nn-static_${cann_version}_linux-${arch}.tar.gz
    ```

   ${repo_path} represents the project root directory. ${soc_name} represents the NPU model name, which is ${soc_version} with "ascend" removed. After successful compilation, the compressed package is stored in the build_out directory.

2. **Extract the ops-nn static library**

   Enter the build_out directory and execute the extraction command:

    ```bash
   tar -zxvf ./cann-${soc_name}-ops-nn-static_${cann_version}_linux-${arch}.tar.gz -C ${static_lib_path}
    ```

   ${static_lib_path} represents the static library extraction path. The directory structure after extraction is as follows:

    ```text
    ├── cann-${soc_name}-ops-nn-static_${cann_version}_linux-${arch}
    │   ├── lib64
    │   │   ├── libcann_nn_static.a                # Static library file
    │   └── include
    |       ├── ...                                 # aclnn interface header files
    ```

### Offline Compilation

If you need to compile in an environment without internet access, prepare the required third-party software in advance, and then perform source code compilation. The specific process is as follows:

1. **Check whether the basic environment is complete**

    Ensure that the basic environment setup has been completed according to [Environment Deployment](../context/quick_install.md), including CANN package installation and source code download.

    - In a networked environment, go to [this project homepage](https://gitcode.com/cann/ops-nn) and download the source code through the `Download ZIP` or `clone` button.
    - Connect to the offline environment and upload the source code to your specified directory. If you downloaded a source code compressed package, extract it first.

2. **Download third-party software dependencies**

    Download the third-party software in a networked environment in advance. Currently, the following methods are available; choose as needed:

    - Method 1: Manually download according to the table provided in [Third-Party Software Dependencies](#third-party-software-dependencies). If downloading from other addresses, ensure the version numbers are consistent.

    - Method 2: Use the [third_lib_download.py](../../../scripts/tools/third_lib_download.py) script for one-click download. This script is in the `scripts/tools/` directory of this project. Download the script and execute the following command:

        ```bash
        python ${scripts_dir}/third_lib_download.py
        ```

    ${scripts_dir} represents the script storage path. The downloaded third-party software packages are stored by default in the directory where the script is located.

3. **Compile the operator package**

    Upload the downloaded third-party software to the offline environment. You can store it in the `third_party` directory or a custom directory. **The former is recommended because the compilation commands are the same as those in the online compilation scenario.**

    - **third_party directory** (Recommended)

        Create a `third_party` directory in the project root directory (if it already exists, no need to create it), and copy the third-party software to this specified directory. The compilation commands are the same as those for online compilation. For details, refer to [Online Compilation](#online-compilation).

    - **Custom directory**

        Create a ${cann_3rd_lib_path} directory in any location in the offline environment, and copy the third-party software to this directory. Ensure that this directory has access permissions.

        ```bash
        mkdir -p ${cann_3rd_lib_path}
        ```

        In this case, the compilation command requires an additional `--cann_3rd_lib_path=${cann_3rd_lib_path}` parameter compared to the online compilation command to specify the third-party software path. Assuming the storage path is `/path/cann_3rd_lib_path`, the commands for different compilation methods are as follows:

      - Custom operator package

        ```bash
        bash build.sh --pkg --soc=${soc_version} [--vendor_name=${vendor_name}] [--ops=${op_list}] --cann_3rd_lib_path=${cann_3rd_lib_path}
        # Example: Compile the TransposeBatchMatMul operator
        # bash build.sh --pkg --soc=ascend910b --ops=transpose_batch_mat_mul --cann_3rd_lib_path=/path/cann_3rd_lib_path
        ```

      - ops-nn full package

        ```bash
        bash build.sh --pkg --soc=${soc_version} --cann_3rd_lib_path=${cann_3rd_lib_path}
        # bash build.sh --pkg --soc=ascend910b --cann_3rd_lib_path=/path/cann_3rd_lib_path
        ```

      - ops-nn static library

        ```bash
        bash build.sh --pkg --static --soc=${soc_version} --cann_3rd_lib_path=${cann_3rd_lib_path}
        # bash build.sh --pkg --static --soc=ascend910b --cann_3rd_lib_path=/path/cann_3rd_lib_path
        ```

4. **Install/Uninstall the operator package**

    The operator package results compiled in offline and online scenarios are the same. They are stored by default in the build_out directory of the project root directory, and the installation and uninstallation commands are the same. For details, refer to [Online Compilation](#online-compilation).

## Local Verification

Run operators and UT test cases through the build.sh script in the project root directory. Currently, operators support API mode (aclnn interface) and graph mode invocation. **aclnn invocation is recommended.**

### Run Operator Samples

> **Note**: For Ascend 950PR products, use simulation to run operator samples. For details, refer to [Simulation Guide](../debug/op_debug_prof.md#method-2-for-ascend-950pr).

- Run operator samples based on a **custom operator package**. After the package is installed, execute the following command:

    ```bash
    bash build.sh --run_example ${op} ${mode} ${pkg_mode} [--example_name=${example_name}] [--vendor_name=${vendor_name}] [--soc=${soc_version}] [--simulator]
    # Example: Run test_aclnn_transpose_batch_mat_mul.cpp for the TransposeBatchMatMul operator
    # bash build.sh --run_example transpose_batch_mat_mul eager cust --example_name=transpose_batch_mat_mul --vendor_name=transpose_batch_mat_mul
    ```

    - ${op}: Represents the operator to run. The operator name is in lowercase underscore format, such as transpose_batch_mat_mul.
    - ${mode}: Represents the execution mode. Currently, eager (aclnn invocation) and graph (graph mode invocation) are supported.
    - ${pkg_mode}: Represents the package mode. Currently, only cust (custom operator package) is supported.
    - ${example_name} (optional): Represents the sample name to run. The name is the file name in each operator's examples folder, with the `test_aclnn_` prefix and `.cpp` suffix removed.
    - ${vendor_name} (optional): Consistent with the custom operator package build settings. The default name is custom.
    - ${soc_version} (optional): Represents the NPU model. When set to "ascend950", the sample files in the "arch35" directory are also run.
    - ${simulator} (optional): Represents the simulation mode. Currently, it is only supported in the eager (aclnn invocation) scenario. In simulation mode, the corresponding simulation library is linked based on soc_version.

    Note: When ${mode} is graph, ${pkg_mode} and ${vendor_name} do not need to be specified.

- Run operator samples based on an **ops-nn package**. After installation, execute the following command:

    ```bash
    bash build.sh --run_example ${op} ${mode} [--soc=${soc_version}] [--simulator]
    # Example: Run the TransposeBatchMatMul operator sample
    # bash build.sh --run_example transpose_batch_mat_mul eager
    ```

    - ${op}: Represents the operator to run. The operator name is in lowercase underscore format, such as transpose_batch_mat_mul.
    - ${mode}: Represents the operator execution mode. Currently, eager (aclnn invocation) and graph (graph mode invocation) are supported.
    - ${soc_version} (optional): Represents the NPU model. When set to "ascend950", the sample files in the "arch35" directory are also run.
    - ${simulator} (optional): Represents the simulation mode. Currently, it is only supported in the eager (aclnn invocation) scenario. In simulation mode, the corresponding simulation library is linked based on soc_version.

- Run operator samples based on an **ops-nn static library**:

    1. **Prerequisites**

        The ops-nn static library depends on the ops-legacy static library and the ops-math static library. Prepare the above static libraries, extract them, and move all lib64 and include directories to a unified ${static_lib_path} directory.

        > Note: The ops-legacy static library `cann-${soc_name}-ops-legacy-static_${cann_version}_linux-${arch}.tar.gz` can be obtained by clicking the [download link](https://ascend.devcloud.huaweicloud.com/artifactory/cann-run-release/software/master). The ops-nn static library and ops-math static library packages are not yet available; generate them through local compilation.

    2. **Create run.sh**

        Create a run.sh file in the same directory as the operator's `examples/test_aclnn_${op_name}.cpp` file.

        The following example uses the TransposeBatchMatMul operator running test_aclnn_transpose_batch_mat_mul.cpp:

        ```bash
        # Activate environment variables
        if [ -n "$ASCEND_INSTALL_PATH" ]; then
            _ASCEND_INSTALL_PATH=$ASCEND_INSTALL_PATH
        elif [ -n "$ASCEND_HOME_PATH" ]; then
            _ASCEND_INSTALL_PATH=$ASCEND_HOME_PATH
        else
            _ASCEND_INSTALL_PATH="/usr/local/Ascend/cann"
        fi

        source ${_ASCEND_INSTALL_PATH}/bin/setenv.bash

        # Compile the executable file
        g++ test_aclnn_transpose_batch_mat_mul.cpp -I ${static_lib_path}/include -L ${static_lib_path}/lib64 -L ${ASCEND_HOME_PATH}/lib64 -Wl,--allow-multiple-definition \
            -Wl,--start-group -lcann_nn_static -lcann_math_static -lcann_legacy_static -Wl,--end-group -lgraph -lgraph_base \
            -lpthread -lmmpa -lmetadef -lascendalog -lregister -lopp_registry -lops_base -lascendcl -ltiling_api -lplatform \
            -ldl -lc_sec -lnnopbase -lruntime -lerror_manager -lunified_dlog -o test_aclnn_transpose_batch_mat_mul   # Replace with the actual operator executable file name

        # Execute the program
        ./test_aclnn_transpose_batch_mat_mul
        ```

        ${static_lib_path} represents the unified static library storage path. ${ASCEND_HOME_PATH} is configured through the environment variable and represents the CANN toolkit package installation path, typically ${install_path}/cann. Replace the final executable file name with the actual operator executable file name.
        lcann_nn_static, lcann_math_static, and lcann_legacy_static represent the static library files that the operator depends on, obtained from the unified static library storage path ${static_lib_path}. lgraph, lmetadef, and others represent the underlying library files that the operator depends on, which can be obtained from the CANN toolkit package.

    3. **Execute run.sh**

        ```bash
        bash run.sh
        ```

Regardless of the method used above, the operator sample execution result is printed. The following example uses the TransposeBatchMatMul operator:

```text
result[0] is: 0.000000
result[1] is: 0.000000
result[2] is: 0.000000
result[3] is: 0.000000
...
```

### Run Operator UT

> Note: Running UT test cases depends on the googletest unit testing framework. For detailed information, refer to the [googletest official website](https://google.github.io/googletest/advanced.html#running-a-subset-of-the-tests).

```bash
# Install the test-related requirements.txt dependencies in the root directory
pip3 install -r tests/requirements.txt
# Method 1: Compile and run UT test cases for the specified operator and corresponding function (choose one)
bash build.sh -u --[opapi|ophost|opkernel] --ops=transpose_batch_mat_mul
# Method 2: Compile and run all UT test cases
# bash build.sh -u
# Method 3: Compile all UT test cases without running them
# bash build.sh -u --noexec
# Method 4: Compile and run UT test cases for the corresponding function (choose one)
# bash build.sh -u --[opapi|ophost|opkernel]
# Method 5: Compile UT test cases for the corresponding function without running them (choose one)
# bash build.sh -u --noexec --[opapi|ophost|opkernel]
# Method 6: Specify soc compilation when running UT test cases
# bash build.sh -u --[opapi|ophost|opkernel] [--soc=${soc_version}]
```

Assuming you want to verify whether the ophost function works properly, execute the following command:

```bash
bash build.sh -u --ophost
```

After execution, the following content appears, indicating successful execution:

```bash
Global Environment TearDown
[==========] ${n} tests from ${m} test suites ran. (${x} ms total)
[  PASSED  ] ${n} tests.
[100%] Built target nn_op_host_ut
```

${n} represents the number of test cases executed, ${m} represents the number of test items, and ${x} represents the time consumed for executing the test cases, in milliseconds.
